# 회의록: 모바일 영수증 검색 베타 준비

참석자:
- 민아: 제품 리드
- 준호: 백엔드 엔지니어
- 서연: 프론트엔드 엔지니어
- 도윤: 데이터/검색 담당
- 해린: 고객 성공 담당

민아: 오늘은 모바일 영수증 검색 베타를 다음 주에 열 수 있는지 확인하려고 합니다. 전체 출시가 아니라 베타라서, 범위를 작게 잡아야 해요. 고객들이 제일 많이 요청한 건 "지난달 점심 영수증 찾아줘" 같은 검색이에요. 일단 모바일에서 영수증 제목, 금액, 날짜로 검색이 되면 충분하다고 봅니다.

해린: 고객 입장에서는 검색 결과가 너무 많으면 불편해요. 특히 같은 금액의 영수증이 여러 개 있을 때 날짜가 같이 보여야 합니다. 그리고 환불된 영수증은 결과에서 제외해달라는 요청이 있었어요. 다만 환불 처리 상태가 항상 정확히 들어오는지는 모르겠습니다.

준호: 백엔드에는 이미 receipts 테이블이 있고 refunded_at 컬럼도 있습니다. 그런데 일부 오래된 데이터는 refunded_at이 비어 있는데 status가 refunded로 들어가 있을 수 있어요. 제가 오늘 안에 샘플 쿼리를 확인해보겠습니다. 금액 검색은 정확히 일치만 할지, 범위 검색까지 할지는 아직 정해야 합니다.

도윤: 검색 품질 얘기를 하면, 제목 검색은 부분 문자열로 시작해도 됩니다. 형태소 분석이나 벡터 검색은 이번 베타에 필요 없다고 봐요. 다만 "점심"이라고 검색했을 때 title에 lunch가 들어간 영수증을 찾고 싶다는 이야기가 있었는데, 그건 동의어 처리가 필요해서 이번에는 빼는 게 맞습니다.

서연: 프론트에서는 검색창 하나와 결과 리스트 정도면 됩니다. 결과 카드에는 제목, 금액, 날짜, 가맹점 이름을 보여주겠습니다. 환불 제외 토글을 넣자는 의견이 있었는데, 기본값을 제외로 하고 토글은 나중에 넣어도 될 것 같아요.

민아: 저는 베타에서는 환불된 영수증이 보이면 안 된다고 생각해요. 토글은 오히려 혼란스러울 수 있습니다. 그래서 기본적으로 제외하고, 왜 안 보이는지 설명은 고객지원 문서에서 다루면 됩니다.

해린: 그런데 고객지원팀은 환불된 영수증도 가끔 확인해야 합니다. 베타 사용자에게는 숨기더라도 내부 CS 모드에서는 볼 수 있어야 하지 않을까요?

준호: 내부 CS 모드는 권한도 들어가고 범위가 커집니다. 이번 주에는 하지 않는 게 좋겠습니다. 대신 제가 API 응답에 refunded 여부가 섞이지 않도록 필터 조건을 명확히 하겠습니다. 금요일 오전까지 백엔드 필터 PR을 올리겠습니다.

도윤: 검색 로그도 필요합니다. 어떤 검색어에서 결과가 0개인지 봐야 다음 개선을 할 수 있어요. 하지만 개인정보 이슈가 있으니까 검색어 원문을 그대로 저장하면 안 됩니다. 저는 이번 베타에서는 로그를 안 넣고, 다음 스프린트에서 익명화 설계를 먼저 하는 게 맞다고 봅니다.

민아: 베타 오픈 전에 최소한 수동 체크리스트는 있어야 해요. "제목으로 검색", "금액으로 검색", "날짜로 검색", "환불 영수증 제외" 네 가지는 확인해야 합니다. 서연님이 목요일 오후까지 프론트 화면을 붙이고, 금요일에는 같이 스모크 테스트를 합시다.

서연: 네, 목요일 오후까지 검색창과 결과 리스트를 붙이겠습니다. 다만 날짜 검색 입력 형식을 YYYY-MM-DD로 할지, 달력 컴포넌트를 쓸지는 정해야 합니다. 실제 베타도 우선 텍스트 입력으로 시작할 수 있습니다.

해린: 고객 안내 문구도 필요합니다. "현재는 제목, 금액, 날짜 기준으로 검색됩니다. 환불된 영수증은 표시되지 않습니다." 이런 식이면 될까요? 제가 초안을 작성하겠습니다. 마감은 금요일 점심 전까지로 하겠습니다.

민아: 좋습니다. 그리고 금액 범위 검색은 이번 베타에서 제외합니다. 정확히 일치만 해요. 동의어 검색, CS 모드, 검색 로그도 이번에는 제외입니다. 대신 이 네 개는 다음 단계 후보로 기록해 주세요.

도윤: 그러면 제가 다음 스프린트 후보 목록에 동의어 검색, 익명화된 검색 로그, CS 모드, 금액 범위 검색을 적어두겠습니다. 오늘 회의 후에 이슈 초안으로 남기겠습니다.

준호: 한 가지 애매한 게 있습니다. "지난달" 같은 자연어 날짜 검색은 이번 범위인가요? 만약 사용자가 지난달이라고 치면 날짜 범위로 해석해야 하는데, 그건 별도 로직이 필요합니다.

민아: 아니요. 이번에는 자연어 날짜 검색은 하지 않습니다. 사용자가 날짜를 직접 입력하는 것으로 제한합시다. 이 부분은 고객 안내 문구에 포함해 주세요.

해린: 알겠습니다. 제가 안내 문구에 날짜 직접 입력 제한도 넣겠습니다.

민아: 정리하면, 이번 베타의 필수 범위는 모바일 검색창, 제목/금액/날짜 검색, 결과 리스트, 환불 영수증 제외, 그리고 수동 스모크 체크리스트입니다. 각자 액션 아이템을 정리해서 공유해 주세요.
